# v0.4 Pre-release Acceptance

日期：2026-06-19

## 结论

v0.4 当前应定义为 Aether_island 的“基础壳与运行时预发布版本”，不是完整产品版。

本版目标是证明三件事：

1. Windows 原生透明岛窗口能稳定启动并被探针验证。
2. Rust-first widget runtime 能把多个组件统一调度并推送给前端。
3. Python sidecar 的协议边界已经预留并有测试，但暂不由 Rust 正式拉起。

这个定义可以避免把最初 IslandGate 的完整设想全部压进 v0.4，导致版本边界失控。

## 当前版本边界

### v0.4 包含

- Rust-first Tauri 主体。
- Windows 原生窗口能力验证。
- 透明、置顶、无边框岛窗口。
- Rust widget runtime 核心抽象。
- System Clock、Mock、Sidecar 三个示例 widget。
- 前端 snapshot 渲染与 widget 切换。
- Python sidecar stdio JSON 协议草稿。
- 统一质量门禁与窗口烟测。
- 外部技术笔记持续更新。

### v0.4 不包含

- 真正的 Python 子进程拉起、重启、退出回收。
- GitHub GraphQL、番茄钟、剪贴板、脚本监控等真实插件。
- 系统托盘菜单。
- 开机自启动。
- 安装包签名、更新器、发布流水线。
- 跨平台窗口等价实现。

这些不是被忘了，而是当前版本刻意延期。原因是 v0.4 的核心风险在窗口壳、runtime 契约和本地验证；如果此时把 tray、autostart、插件引擎、安装器一起塞进来，调试边界会变得很散。

## 小白解释

你现在做的不是“完整成品岛”，而是先做“能稳定站住的岛地基”。

可以这样理解：

```text
v0.4 = 窗口壳 + Rust 运行时 + 前端显示协议 + Python 接口草图 + 验证脚本
```

而不是：

```text
v0.4 = 所有插件 + 托盘 + 自启动 + 打包发布 + 完整双进程
```

这个版本如果收得好，后面加小组件会更顺；如果这个版本边界不清，后面每加一个功能都会和窗口、runtime、IPC、UI 混在一起乱改。

## Rust 知识点

### 1. 为什么先做 Rust runtime

Rust runtime 负责的是“程序内部可靠调度”：

- 哪些 widget 已注册。
- 哪个 widget 优先级最高。
- 用户主动选择了哪个 widget。
- widget 刷新失败后如何转成 error snapshot。
- 退出时如何广播 shutdown。

这些逻辑放在 Rust 里，编译器能帮你检查类型边界。比如 `WidgetTone` 是枚举，不是随手拼出来的字符串；`WidgetSnapshot` 的字段缺失会直接编译失败。

### 2. 为什么 Python 先只做协议草稿

Python 很适合后续做：

- GitHub API 抓取。
- 本地脚本状态读取。
- 自动化任务状态聚合。
- 剪贴板、文件、日志等生态集成。

但是 Python 子进程生命周期管理是另一套复杂度，包括：

- 如何启动。
- 如何判断 ready。
- stdout/stderr 如何读。
- Rust 退出时如何关闭 Python。
- Python 卡死时如何重启。
- JSON 消息乱序或坏包如何处理。

所以 v0.4 只固定协议草稿，先不把生命周期复杂度引进来，是合理的。

## 验收命令

在 `D:\Aether_island` 运行：

```powershell
pnpm check
pnpm smoke:window
```

验收标准：

- `pnpm check` 通过 Rust format/test/clippy、前端 build、Python ruff/pytest。
- `pnpm smoke:window` 输出 `Aether_island window assertions passed.` 与 `Window smoke passed.`。

## 预发布风险

当前仍有这些风险，但不阻止 v0.4 基础版收口：

1. 鼠标穿透的真实交互体验还需要人工长时间使用验证。
2. `SidecarWidget` 还不是实际 Python 子进程管理器。
3. 没有 tray/autostart，所以还不是日常常驻应用。
4. 目前是 Windows-first，跨平台只保留技术可能性，没有做等价实现。
5. README 已说明 v0.4 范围，但还没有正式 changelog 和 release tag。

## 下一步建议

如果当前两条验收命令持续通过，可以把 v0.4 进入“收口阶段”：

1. 做一次代码审查，重点看未处理错误、线程退出、窗口脚本和接口契约。
2. 补一份 `CHANGELOG.md` 或 v0.4 release note。
3. 在 Git 中整理本阶段改动，准备一次清晰提交。
4. v0.5 再进入系统托盘、Rust 启动 Python sidecar、真实插件之一。
