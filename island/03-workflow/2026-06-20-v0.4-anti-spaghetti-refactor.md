# Aether_island v0.4 反屎山重构记录

## 结论

本次重构目标不是增加新功能，而是降低后续添加小组件、Python sidecar、GitHub 热力图时的结构风险。

核心处理方向：

- 前端从 `main.ts` 大入口拆成“契约、视图、Tauri 桥接、预览数据、启动编排”。
- Rust 从 `lib.rs` 大入口拆成“应用状态、命令、窗口策略、后台任务、核心 runtime”。
- 修复 Rust 命令中“持有 runtime 锁时发前端事件”的架构风险。
- 将 widget 刷新流程拆成“收集刷新结果、选择快照、输出事件”。
- 为 `HoverController` 增加 `disconnect()`，并抽出纯几何函数 `containsPoint()`。

## 为什么要这样拆

### 1. `main.ts` 不应该知道所有事情

重构前，`main.ts` 同时负责：

- 类型定义。
- 数据校验。
- DOM 查询。
- DOM 渲染。
- Tauri IPC。
- 浏览器预览数据。
- 窗口定位。
- hover 控制器初始化。

这会导致后续每做一个小组件，都可能改到入口文件。入口文件一旦长期膨胀，最容易变成无法判断影响范围的“总控脚本”。

现在的拆分边界：

- `contracts.ts`：前后端数据契约和运行时校验。
- `island-view.ts`：DOM 渲染和按钮绑定。
- `tauri-bridge.ts`：Tauri invoke/listen/window API。
- `preview-data.ts`：浏览器模式 mock 数据。
- `main.ts`：只负责启动编排。

### 2. Rust `lib.rs` 不应该承担架构主干所有职责

重构前，`lib.rs` 同时负责：

- `RuntimeState` 状态定义。
- 点击穿透策略。
- Tauri command。
- 后台刷新线程。
- 前端事件发送。
- 窗口 Win32 样式初始化。
- Tauri builder 启动。

现在的拆分边界：

- `app_state.rs`：全局状态和点击穿透策略枚举。
- `commands.rs`：前端能调用的 Tauri command。
- `runtime_tasks.rs`：启动快照、刷新循环、退出清理、前端事件发送。
- `shell_window_policy.rs`：窗口壳层策略，包括 Win32 样式和点击穿透状态切换。
- `lib.rs`：Tauri 应用启动骨架。

### 3. 锁内发事件是隐性风险

Rust command 里不能长期持有 `Mutex<NativeRuntime>` 后再做跨边界行为。

更稳的模式是：

```rust
let events = {
    let mut runtime = runtime_state
        .runtime
        .lock()
        .map_err(|_| "runtime state lock is poisoned".to_string())?;

    runtime.handle_shell_state(shell_state)
};

emit_frontend_events(&app_handle, events);
```

关键点：

- `{ ... }` 代码块结束后，锁会释放。
- 事件发送发生在锁外。
- 后续即使前端事件触发新的 command，也更不容易出现锁等待链条。

## 后续建议

下一阶段不要急着加复杂功能，建议先做一个真实但小的 widget，例如 GitHub Heatmap 的最小只读版。

推荐顺序：

1. 新增一个 Rust widget module，不直接塞进 `widget.rs`。
2. 先接静态数据，确认 UI 展示 contract。
3. 再决定 GitHub 数据由 Rust 拉，还是走 Python sidecar。
4. 如果开始接 Python sidecar，再补子进程生命周期测试。

当前结构已经允许继续演进，但 `core/widget.rs` 后续仍建议拆成目录模块：

```text
core/widget/
├── mod.rs
├── mock.rs
├── system_clock.rs
└── sidecar_widget.rs
```

这样每个 widget 不会互相污染。
