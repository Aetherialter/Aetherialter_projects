# Aether_island v0.5 GitHub Heatmap 静态 Widget 实现记录

## 结论

v0.5 已完成 GitHub Heatmap 静态数据版 widget。

本次实现没有引入真实 GitHub API、网络请求、Token、缓存或 Python sidecar。目标是先验证 widget runtime 的扩展边界：新增一个 widget 是否能保持低耦合、少改动、可测试。

## 防屎山措施

### 1. 不把 GitHub 逻辑塞进 provider

`NativeWidgetProvider` 只负责注册：

```rust
provider.register(create_github_heatmap_widget());
```

它不知道 GitHub 的数据结构，也不负责摘要格式化。

### 2. 不让 `widget.rs` 继续膨胀

原来的 `widget.rs` 已拆成目录模块。后续新增 widget 时，只需要新增文件并在 `mod.rs` 导出工厂函数。

### 3. 静态数据与展示格式分离

GitHub widget 内部拆成：

- `GithubHeatmapSummary`：数据结构。
- `static_heatmap_summary()`：静态数据来源。
- `format_github_summary()`：摘要格式化。
- `snapshot()`：转换成前端 snapshot。

### 4. 测试覆盖新增 widget

新增 Rust 单元测试：

- 静态 GitHub snapshot 字段正确。
- 静态数据刷新间隔正确。

## v0.6 建议

v0.6 可以开始把 GitHub Heatmap 从静态数据升级为可替换数据源，但仍然不要直接在 widget 内写死 HTTP。

推荐引入组合式数据源接口：

```rust
pub trait GithubContributionSource: Send {
    fn load_summary(&mut self) -> WidgetResult<GithubHeatmapSummary>;
}
```

然后逐步替换为：

- `StaticGithubContributionSource`
- `CacheFirstGithubContributionSource`
- `PythonSidecarGithubContributionSource`
- `HttpGithubContributionSource`

这样 widget 主体不需要因为数据来源变化而反复改动。
